# EduLink PRD v2.0 업그레이드 마스터 개발 계획

기준일: 2026-03-24  
기준 문서: Edulink_PRD_v2.0.md, DEV_PLAN.md, docs/development-progress.md  
전제: 현재 v1.0 서비스는 유지하고, v2.0은 병행 개발 후 점진 전환한다.

## 1. 계획 목적

이 계획의 목표는 다음 3가지를 동시에 달성하는 것이다.

1. 현재 v1.0 기능과 운영 흐름을 깨지 않고 유지한다.
2. PRD v2.0의 방향인 Education Workforce OS로 제품을 구조적으로 확장한다.
3. 어느 시점에서도 v1.0으로 즉시 되돌릴 수 있는 롤백 장치를 마련한다.

즉, 이번 업그레이드는 기존 제품을 갈아엎는 재개발이 아니라, v1.0을 안정된 기준선으로 둔 상태에서 v2.0 기능을 병렬로 올리는 방식으로 진행한다.

---

## 2. 현재 상태 요약

### 2.1 유지해야 할 v1.0 자산
- 인증, 권한, 기본 회원가입/로그인
- 공고 CRUD, 지원, 승인/거절 흐름
- 세션, 정산, 세무, 리뷰, SOS, 계약, 관리자/교육청 API의 기본 골격
- 단일 프론트엔드 쉘 기반 역할별 UI
- 상용 배포 흐름(Cloudflare Worker + 별도 백엔드)
- 정책/신뢰 페이지와 공개 자산

### 2.2 v2.0에서 바뀌는 방향
- 강사 매칭 중심 MVP에서 기관 운영 중심 인프라로 무게 중심 이동
- 단건 기능 나열형 UI에서 운영 레이어 중심 UX로 재구성
- 정산 규칙을 계약 유형 기반으로 고도화
- 공급자 데이터(Supply Cloud), 운영(Operations Cloud), 재무/컴플라이언스(Finance & Compliance Cloud)로 도메인 분리
- 리퍼럴/제휴 기반 성장 모델로 수익 구조 전환

### 2.3 핵심 전환 원칙
- v1.0 라인은 항상 배포 가능한 상태로 유지한다.
- 데이터 스키마는 expand -> migrate -> contract 순서로 변경한다.
- 새 기능은 기능 플래그 뒤에 두고 기본값은 off로 시작한다.
- URL, API 계약, 운영자 툴은 가급적 하위 호환을 유지한다.
- 전환은 역할 단위가 아니라 기능 단위로 점진 적용한다.

## 3. 목표 아키텍처와 전환 방식

## 3.1 목표 제품 구조

### Supply Cloud
- 강사 프로필
- 자격/서류 검증
- 평판/리뷰/재배정률
- 가용 슬롯, 활동 이력, 자격 상태

### Operations Cloud
- 기관 공고 관리
- 지원자 비교 및 배치
- 세션/출결/수업일지
- 반복 수업 배정, 커뮤니케이션, SOS 대응

### Finance & Compliance Cloud
- 계약 유형별 정산 엔진
- 원천징수/지급명세/환불
- 감사 로그
- 민감정보 보호 및 접근 통제

## 3.2 전환 전략

단일 big-bang 교체는 금지한다. 아래 3개 레이어를 분리해서 올린다.

1. 도메인/데이터 레이어 선반영
2. API 및 내부 서비스 레이어 이중화
3. 프론트엔드 화면과 사용자 흐름 점진 전환

이 방식이면 화면은 v1.0이어도 데이터 구조와 내부 서비스는 v2.0 준비를 먼저 끝낼 수 있다.

---

## 4. v1.0 유지 전략

## 4.1 운영 기준선 정의

v1.0 기준선은 아래를 의미한다.
- backend/app_jwt.py 기반 현재 API 집합
- frontend/index.html 기반 현재 사용자 흐름
- 현재 QA 계정, 회귀 테스트, 정책 페이지, 배포 방식

## 4.2 기준선 고정 장치

반드시 먼저 수행한다.

1. `release/v1-stable` 브랜치 생성
2. 현재 운영 가능한 시점을 `v1.0-stable` 태그로 고정
3. 현재 SQLite 스키마 스냅샷 보관
4. `dist/` 및 Worker 배포 버전 기록
5. 회귀 테스트 결과를 v1 기준선으로 보관

## 4.3 병행 운영 방식

- `main`: v2 통합 개발 브랜치
- `release/v1-stable`: 긴급 운영 안정화 전용
- 기능 플래그 off 상태에서는 항상 v1 동작 유지
- v2 실험 기능은 운영자 또는 특정 QA 계정에만 노출

---

## 5. 롤백 장치 설계

## 5.1 롤백 원칙

롤백은 코드만 되돌리는 것이 아니라 아래 4개를 동시에 복구 가능해야 한다.

1. 프론트엔드 노출 상태
2. API 라우팅/서비스 동작
3. 데이터 스키마와 마이그레이션 상태
4. 운영 설정값과 기능 플래그

## 5.2 필수 롤백 장치

### A. Git 기준선
- `v1.0-stable` 태그 유지
- `release/v1-stable` 브랜치 유지
- v2 릴리스 직전마다 `pre-v2-cutover-*` 태그 생성

### B. 기능 플래그
최소 아래 플래그를 둔다.
- `FF_V2_SUPPLY_CLOUD`
- `FF_V2_OPERATIONS_CLOUD`
- `FF_V2_FINANCE_ENGINE`
- `FF_V2_ROLE_HOME`
- `FF_V2_CONTRACT_PRICING`
- `FF_V2_MATCHING_ENGINE`

모든 플래그 기본값은 `false`로 시작한다.

### C. 데이터베이스 전환 규칙
- 컬럼 추가는 허용, 기존 컬럼 삭제는 마지막 단계까지 금지
- 새 테이블은 기존 기능과 독립적으로 추가
- 데이터 백필 스크립트는 idempotent하게 작성
- 구 스키마를 참조하는 v1 코드가 계속 동작하도록 유지

### D. 배포 롤백
- Worker는 이전 Version ID 기록 및 즉시 재배포 가능 상태 유지
- 백엔드 환경변수 세트는 `v1`, `v2-canary`, `v2-prod`로 분리
- 운영 URL 전환은 기능 플래그로 처리하고 URL 자체는 늦게 바꾼다.

## 5.3 실제 롤백 절차

### 레벨 1. 화면만 롤백
- 증상: v2 화면 UX 문제, 기능은 정상
- 조치: `FF_V2_ROLE_HOME=false`, `FF_V2_OPERATIONS_CLOUD=false`
- 기대 결과: 데이터와 API는 유지한 채 v1 화면으로 즉시 복귀

### 레벨 2. API 동작 롤백
- 증상: 신규 도메인 서비스 로직 오류
- 조치: 신규 서비스 경로 비활성화, 기존 app_jwt 흐름으로 스위치
- 기대 결과: DB 확장 상태는 유지하되 요청 처리는 v1 API 로직으로 회귀

### 레벨 3. 배포 버전 롤백
- 증상: Worker 또는 앱 버전 전체 장애
- 조치: 이전 Worker Version ID 재적용, 백엔드 이미지/커밋을 `v1.0-stable`로 복귀
- 기대 결과: 배포 단위 전체 복원

### 레벨 4. 데이터 롤백
- 증상: 마이그레이션 오류나 데이터 오염
- 조치: 사전 백업 DB 복원 + 백필 스크립트 차단
- 기대 결과: 스키마와 데이터 모두 v1 기준으로 복원

## 6. 전체 개발 단계

전체 일정은 6개 Program, 18개 Sprint로 나눈다.  
각 Sprint는 1주 기준이며, 안정화 Sprint는 기능 추가보다 품질 게이트에 집중한다.

## Program A. 기준선 고정과 전환 기반 구축

### Sprint A1. v1 기준선 봉인
목표:
- 현재 v1 운영 상태를 기술적으로 봉인한다.

작업:
- `release/v1-stable` 브랜치 생성
- `v1.0-stable` 태그 생성
- DB 스냅샷, 배포 버전, QA 계정, 테스트 결과 기록
- 배포/롤백 런북 초안 작성

완료 기준:
- 30분 내 v1 배포 복원이 가능한 상태

### Sprint A2. 기능 플래그와 환경 분리
목표:
- v2 기능을 끄고 켤 수 있는 운영 장치를 만든다.

작업:
- 프론트엔드/백엔드 공통 기능 플래그 설계
- dev, canary, prod 환경 분리
- 주요 API 경로별 신규/기존 핸들러 전환 포인트 정리

완료 기준:
- 코드 배포 없이 설정값만으로 v2 기능 비활성화 가능

### Sprint A3. 데이터 전환 프레임 준비
목표:
- v2 데이터 구조 확장을 시작할 수 있게 한다.

작업:
- 마이그레이션 정책 수립
- 백필 스크립트 템플릿 작성
- 감사 로그, 계약 유형, 공급자 메타데이터용 신규 테이블 설계

완료 기준:
- 첫 번째 v2 마이그레이션이 v1 기능을 깨지 않고 적용 가능

## Program B. Supply Cloud 구축

### Sprint B1. 강사 프로필 정규화
목표:
- v1 프로필/리뷰/활동 데이터를 강사 마스터 데이터로 승격한다.

작업:
- 강사 기본정보, 자격, 서류, 검증상태 스키마 정리
- 프로필 완성도 점수
- 서류 만료일, 검증 이력, 제재 상태 추가

완료 기준:
- 강사 1명에 대해 프로필, 서류, 평판, 검증 상태를 단일 조회로 표현 가능

### Sprint B2. 공급자 평판 엔진
목표:
- 강사 선발에 필요한 신뢰 지표를 만든다.

작업:
- 재배정률, 응답속도, 노쇼율, 리뷰 평균, 수업 완료율 집계
- 공급자 점수 계산식 설계
- 관리자 보정 기능

완료 기준:
- 기관 화면에서 강사 비교 시 점수와 근거 지표 확인 가능

### Sprint B3. 가용 슬롯과 배치 가능성 모델
목표:
- 강사 검색이 단순 검색이 아닌 배정 가능성 중심으로 개선된다.

작업:
- 가용 시간 캘린더 저장 구조 재정비
- 지역/시간/과목 적합도 계산
- 반복 수업용 예약 가능성 지표 추가

완료 기준:
- 공고 상세에서 배정 가능 강사 후보가 우선순위로 노출됨

---

## Program C. Operations Cloud 구축

### Sprint C1. 기관 공고 운영 구조 재설계
목표:
- 단일 공고 등록 중심에서 운영 템플릿 중심으로 전환한다.

작업:
- 공고 템플릿
- 반복 공고 생성
- 기관별 표준 필수 입력 항목
- 공고 상태 머신 정비

완료 기준:
- 기관이 반복 업무를 템플릿 기반으로 3분 내 생성 가능

### Sprint C2. 지원자 비교/심사 보드 고도화
목표:
- 기관 운영자가 여러 지원자를 비교하고 빠르게 배정할 수 있게 한다.

작업:
- 비교표 고도화
- 심사 사유 템플릿
- 승인/보류/거절 배치 액션
- 서류/리뷰/가용시간 통합 보기

완료 기준:
- 기관 한 화면에서 심사 완료 가능

### Sprint C3. 세션 운영 레이어 표준화
목표:
- 실제 수업 운영을 제품의 중심 흐름으로 만든다.

작업:
- 출결, 지각, 보정, 일지, 증빙 업로드 정리
- 반복 세션 생성
- 대체강사 재배치 흐름 추가

완료 기준:
- 한 세션의 생성부터 완료까지 운영 기록이 일관되게 남는다.

### Sprint C4. SOS 및 즉시 대체 인력 배치
목표:
- v2의 운영 인프라 특성을 가장 직접적으로 보여준다.

작업:
- SOS 상태 머신
- 알림 채널 정리(SSE 우선)
- 수락 타임아웃, 재배정 로직
- 기관/강사 상태판

완료 기준:
- 긴급 공고 생성 후 후보 노출, 수락, 대체 배정까지 실시간으로 추적 가능

## Program D. Finance & Compliance Cloud 구축

### Sprint D1. 계약 유형 기반 정산 엔진 1차
목표:
- v1의 단일 정산 계산에서 계약 유형 기반 엔진으로 전환한다.

작업:
- 계약 유형 테이블
- 계약별 계산식 버전 관리
- 세션/계약/정산 레코드 연결

완료 기준:
- 동일 세션이라도 계약 유형에 따라 서로 다른 계산 가능

### Sprint D2. 지급/환불/원천징수 레저 정리
목표:
- 재무 데이터를 회계성 이벤트 기준으로 정리한다.

작업:
- 지급 이벤트, 환불 이벤트, 수수료 이벤트 분리
- 정산 상세 레저 생성
- 감사 추적 ID 부여

완료 기준:
- 정산 오류 발생 시 이벤트 단위 역추적 가능

### Sprint D3. 컴플라이언스 및 민감정보 통제
목표:
- 법규 대응 수준을 운영 시스템에 녹인다.

작업:
- 접근 권한 계층화
- 감사 로그 표준화
- 개인정보 마스킹 정책
- 보존기간/파기 정책 연결

완료 기준:
- 운영자 행위와 민감정보 접근이 모두 기록되고 추적 가능

## Program E. v2 사용자 경험 전환

### Sprint E1. v2 역할별 홈 재구성
목표:
- 현재 단일 화면 구조를 유지하되 역할별 첫 화면을 v2 중심으로 재편한다.

작업:
- 강사 홈: 수업/정산/가용시간 중심
- 기관 홈: 공고/심사/세션/지급 중심
- 관리자 홈: 검증/컴플라이언스/관제 중심
- 교육청 홈: KPI/비교/리포트 중심

완료 기준:
- 기능 플래그 on 시 역할별 첫 화면이 v2 홈으로 전환됨

## 13. 최종 판정 기준

v2.0 업그레이드가 완료되었다고 판단하려면 아래를 만족해야 한다.

1. v1.0 기능이 플래그 off 상태에서 완전 유지된다.
2. v2 핵심 기능이 플래그 on 상태에서 카나리 운영을 통과한다.
3. 정산, 세션, 프로필, 리뷰, 계약, 리포트가 v2 도메인 모델에 맞게 연결된다.
4. 운영자가 30분 내 v1.0으로 롤백할 수 있다.
5. 문서, 테스트, 배포, 런북이 동일한 상태를 가리킨다.

이 기준을 충족하지 못하면 v2는 기능이 많아도 아직 완료가 아니다.

## 14. Sprint별 전체 세부 개발계획 (실행표)

아래 실행표는 18개 Sprint를 주차 단위로 분해한 상세 계획이다.  
각 Sprint는 공통적으로 아래 순서로 수행한다.

1. 설계 확정 (0.5일)
2. 구현 (3일)
3. 테스트/보완 (1일)
4. 문서/배포/리허설 (0.5일)

### 공통 산출물 템플릿
- 코드 PR 1개 이상
- 마이그레이션/스크립트 변경 내역
- 테스트 결과 리포트
- 롤백 체크 로그
- 운영 문서 업데이트

### Sprint 1 (A1) - v1 기준선 봉인
목표:
- v1 기준선을 기술적으로 고정한다.

세부 작업:
- Git: `release/v1-stable` 브랜치, `v1.0-stable` 태그 생성
- DB: 현재 SQLite 스키마와 샘플 데이터 덤프 생성
- 운영: Worker 버전, 백엔드 커밋, 환경변수 상태 기록
- 품질: 현재 회귀 테스트 결과 스냅샷 보관

검증:
- 브랜치/태그 기준으로 신규 환경에 v1 재배포 성공

롤백 체크:
- 30분 내 v1 기준선 재배포 리허설 1회

### Sprint 2 (A2) - 기능 플래그/환경 분리
목표:
- v2 기능을 설정 기반으로 제어 가능한 상태로 만든다.

세부 작업:
- 백엔드 플래그 로더 추가
- 프론트엔드 플래그 주입 포인트 추가
- 환경 세트 분리: `v1`, `v2-canary`, `v2-prod`
- 플래그 기본값 전부 off 확정

검증:
- 동일 배포본에서 플래그 on/off에 따라 화면/API 분기 확인

롤백 체크:
- 코드 되돌림 없이 설정값만으로 v1 동작 복귀 확인

### Sprint 3 (A3) - 데이터 전환 프레임
목표:
- v2 데이터 확장을 위한 안전한 마이그레이션 기반을 만든다.

세부 작업:
- expand/migrate/contract 가이드 문서화
- 마이그레이션 네이밍/버전 규칙 확정
- 백필 스크립트 템플릿 + 재실행 안정성 검증
- 감사/정산/공급자 메타 테이블 1차 생성

검증:
- v1 기능 회귀 테스트 전부 통과

롤백 체크:
- 신규 테이블이 있어도 v1 API 문제 없이 동작 확인

### Sprint 4 (B1) - 강사 마스터 프로필 1차
목표:
- 강사 데이터를 단일 마스터 뷰로 조회 가능하게 만든다.

세부 작업:
- 프로필/자격/서류/상태 모델 정규화
- 프로필 완성도 점수 계산 필드 추가
- 관리자 검수 상태 이력 테이블 추가
- 조회 API 통합 엔드포인트 제공

검증:
- 강사 상세 조회 응답에 검증 상태와 서류 상태 포함

롤백 체크:
- 기존 강사 조회 API 응답 스키마 하위호환 유지

### Sprint 5 (B2) - 평판/활동 지표 엔진
목표:
- 기관이 강사를 신뢰 지표 기반으로 비교할 수 있게 한다.

세부 작업:
- 리뷰 평균, 완료율, 노쇼율, 응답속도 집계 파이프라인
- 공급자 신뢰 점수 산식 정의
- 관리자 보정 API 추가
- 지표 캐시/집계 주기 설정

검증:
- 동일 강사에 대해 원시지표와 점수 일관성 확인

롤백 체크:
- 점수 계산 실패 시 v1 리뷰/평점 표시로 자동 대체

### Sprint 6 (B3) - 가용 슬롯/배정 가능성
목표:
- 검색 중심에서 배정 가능성 중심 추천으로 전환 준비.

세부 작업:
- 슬롯 저장 구조 정규화
- 과목/지역/시간 적합도 점수 함수 추가
- 반복 수업 배정 가능성 예측 필드 추가
- 기관 필터 저장 기능 추가

검증:
- 테스트 데이터 기준 우선순위 정렬 일관성 확인

롤백 체크:
- 추천 실패 시 기존 필터 검색 결과로 폴백

### Sprint 7 (C1) - 기관 공고 템플릿화
목표:
- 반복 업무를 줄이는 템플릿 기반 공고 생성 체계 구축.

세부 작업:
- 공고 템플릿 CRUD
- 반복 공고 생성 스케줄 옵션
- 기관별 필수 필드 정책
- 공고 상태 머신(작성/게시/마감/배정완료) 재정의

검증:
- 템플릿으로 공고 생성 시 기존 공고 생성 API와 호환

롤백 체크:
- 템플릿 기능 비활성화 시 기존 공고 폼만 노출

### Sprint 8 (C2) - 지원자 비교/심사 보드
목표:
- 기관 심사 완료 시간을 단축한다.

세부 작업:
- 비교 테이블(경력/리뷰/가용시간/검증상태)
- 일괄 심사 액션(승인/거절/보류)
- 심사 사유 템플릿
- 심사 이력 감사 로그 연결

검증:
- 동일 공고 다중 지원자 심사 E2E 통과

롤백 체크:
- 보드 장애 시 기존 단건 심사 화면으로 우회

### Sprint 9 (C3) - 세션 운영 표준화

세부 작업:

- 지각/노쇼/취소 상태 규칙 정의

미착수/미완료:

## 21. 최종 개발 완료 후 E2E 계획 및 실행 기준

### 21.1 최종 E2E 실행계획

1) 사전 조건
- 백엔드 고정 호스트 또는 안정 터널 활성
- QA 계정 시드 상태 확인
- 브라우저 런타임 설치(Chromium/Firefox)

2) 실행 순서
- Step 1: Chromium 스모크(인증 + 핵심 탭)
- Step 2: Chromium 전체 스위트
- Step 3: Firefox 전체 스위트
